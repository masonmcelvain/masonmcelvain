03 · STARTING POINT

The vendor connector didn’t survive contact

We started with CData Sync, an ETL tool built on NetSuite’s REST API.

SLOW

Record-at-a-time API semantics. Syncs measured in days, not hours.

INCONSISTENT

Missed and duplicated rows undermined faith in the data.

OPAQUE

When it broke, there was nothing to read and nothing to patch. Plus a license bill.

DECISION

Build it in PHP, inside our monolith where cron, ORM, and migrations already live.

05 · FIRST HARD LESSON

Keyset pagination beats OFFSET.

SuiteQL has LIMIT/OFFSET but shifted pages dropped rows.

WHERE lastmodifieddate > :cursor ORDER BY lastmodifieddate, id -- PK tiebreaker, always
  • Idempotent: the 60s overlap re-reads boundaries; replaying a row is safe.
  • Fail loud: report when a whole page shares one timestamp.

07 · THE WALL

~50M rows. 1 week syncs.

Schema evolution was gated on sync speed.

BOTTLENECK 1 — THE SORT

Keyset pagination needs ORDER BY lastmodifieddate — over an unbounded 42M-row set, NetSuite choked on the sort.

BOTTLENECK 2 — THE TRANSPORT

The REST endpoint caps at 1,000 rows per HTTP request. 42M rows = 42,000 sequential round trips, each paying OAuth + HTTP + queueing overhead.

08 · OPTIMIZATION 1 — QUERY SHAPE

Sliding date window: make the sort small

Working theory: SuiteQL applies WHERE before ORDER BY. Bound every query to modifiedDate >= T AND modifiedDate < T + interval and NetSuite sorts a day’s rows — not three years’.

09 · MEASURE, DON’T GUESS

+1 day wins everywhere (in dev)

Interval became a runtime flag (--chunk-interval) to iterate without deploys. Method: 10-minute timed runs, six in parallel per table.

Monotonic across all three tables — up to ~1.8× from a WHERE clause. Strong evidence for the sort-cost theory.

10 · PRODUCTION

Prod disagreed — twice

THE STALL

+1 day froze in prod

Transaction stalled at 24k records for 1.5 hours. Switching to +2 days: 1M records in 10 minutes.

Black-box optimizer, different data distribution. The interval became a per-table knob, tuned empirically — not a global constant.

THE OOM

40M primary keys don’t fit in RAM

Full-sync delete tracking held every seen PK in an in-memory PHP set — OOM at ~40M rows.

Moved tracking to a MySQL temp table: stream PKs in, then DELETE … JOIN the leftovers. And raised the connection timeout to 12h — a silent reconnect would have silently dropped the temp table.

Every fix moved the bottleneck into our own process where we could progressively iterate.

15 · TRADE-OFFS

What I’d defend, and what I’d change

DEFEND

  • PHP + MySQL + cron at 20GB scale. The hard part was the NetSuite connector — which no vendor did well — not orchestration.
  • No retry framework. Overlaps + a 5m schedule: the retry is the schedule.
  • ELT: raw source tables + SQL views.

CHANGE

  • Benchmark in prod conditions sooner — dev extrapolation hid the prod stall.
  • Sync source tables from day one; summary tables were a detour.
  • Checkpoint full syncs per window so failures resume automatically.
